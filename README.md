# Resource-Constrained ReAct Agent

A ReAct (Reason + Act) planning loop agent with hard budget enforcement, three tools, and automatic replanning when stuck.

## Quick Start

```bash
# 1. Clone and enter the repo
git clone https://github.com/FancyCodeMaster/resource-constrained-agent.git
cd resource-constrained-agent

# 2. Copy and fill in your API key
cp .env.example .env
# Edit .env: set OPENAI_API_KEY=sk-...

# 3. Build and run (Docker)
docker build -t react-agent .
docker run --env-file .env react-agent
```

**Single custom task:**
```bash
docker run --env-file .env react-agent python main.py --task "What is the speed of light?"
```

**Specific test task (1–5):**
```bash
docker run --env-file .env react-agent python main.py --task-id 2
```

## Architecture Overview

The system consists of four layers: 
(1) a **Budget Enforcer** that wraps every LLM call and raises a hard `BudgetExceededError` the moment either the 10-call or $0.20 limit is hit, halting execution immediately; 
(2) a **ReAct Agent Loop** that cycles through Think → Act → Observe → Reflect steps, tracking progress and triggering replanning when stuck; 
(3) a **Tool Registry** with three tools (web search, code execution, CSV analysis), each with hard timeouts; 
(4) an **AgentState** dataclass passed through the whole loop that records every step, budget consumption, and the final answer or stop reason.

```
┌─────────────────────────────────────────────────┐
│               Budget Enforcer                   │
│  Tracks: calls_used, cost_used                  │
│  Raises BudgetExceededError on limit            │
└───────────────┬─────────────────────────────────┘
                │
┌───────────────▼─────────────────────────────────┐
│            ReAct Agent Loop                     │
│  1. Think  → LLM picks tool + args              │
│  2. Act    → Execute tool                       │
│  3. Observe → Collect result                    │
│  4. Reflect → "Am I progressing?" (LLM)         │
│     stuck → Replan (LLM call)              │
│  5. Repeat or Final Answer                      │
└───────────────┬─────────────────────────────────┘
                │
┌───────────────▼─────────────────────────────────┐
│             Tool Registry                       │
│  • web_search   (DuckDuckGo, 10s timeout)       │
│  • execute_code (subprocess sandbox, 15s)       │
│  • analyze_csv  (stdlib csv + stats, 10s)       │
└─────────────────────────────────────────────────┘
```

## Planning Loop

**Why ReAct?** 

ReAct interleaves reasoning and acting in a tight loop, making each step's rationale explicit and auditable. This is important for a budget-constrained agent: every LLM call is visible in the trace, making it straightforward to see exactly when the budget was consumed and why. The biggest weakness of ReAct is its **token inefficiency** — each iteration re-sends the full conversation history (all prior thoughts, actions, and observations) to the LLM. In a 10-call budget, this means later calls consume substantially more tokens than early calls, compressing the effective budget and making it harder to complete complex multi-step tasks within cost limits.

## Schema Design

All state is carried in a single `AgentState` dataclass passed through every iteration:

```python
@dataclass
class AgentState:
    task: str                        # original task string
    history: list[Step]              # ordered list of all steps taken
    budget: BudgetState              # live call/cost counters
    stuck_count: int                 # consecutive non-progress steps
    final_answer: str | None         # set when agent finishes
    stopped_reason: str | None       # "answer" | "budget" | "max_iter"

@dataclass
class Step:
    iteration: int
    thought: str                     # LLM's reasoning
    action: str                      # tool name or "final_answer"
    action_input: Any                # argument passed to tool
    observation: str                 # tool's return value (truncated to 2000 chars)
    was_replanned: bool              # True if this step was generated by replan LLM call
    made_progress: bool | None       # result of the reflect call
```

Tools return plain dicts with a consistent `{"success": bool, ..., "error": str | None}` shape, which the dispatcher converts to a single string observation before appending to state. This keeps tool output format-agnostic from the LLM's perspective — the agent always sees a string, never a raw dict.

## Prompt Strategy

Three distinct prompts serve three distinct purposes:

**1. ReAct prompt (`REACT_SYSTEM_PROMPT`)** — tells the LLM its remaining call and cost budget explicitly in the system message every iteration, forces JSON-only output via `response_format={"type": "json_object"}`, lists the three tools with their signatures, and instructs the agent to try a different approach (not retry) when a tool errors. Budget awareness is baked in as a constraint at the top, not a suggestion.

**2. Reflect prompt (`REFLECT_SYSTEM_PROMPT`)** — a lightweight single-purpose call that asks "did the last action move us forward?" and returns `{"made_progress": bool, "reason": "...", "suggestion": "..."}`. Keeping this separate from the main ReAct call means the main reasoning loop stays focused on planning, not self-evaluation.

**3. Replan prompt (`REPLAN_SYSTEM_PROMPT`)** — Provides the full history and explicitly asks for a *completely different* strategy, preventing the agent from re-suggesting what already failed.
